To focus strictly on the OpenShift Platform (Infrastructure) level for a disconnected environment, the chaos experiments must target the "plumbing" that keeps the cluster operational without an external internet connection.
In a disconnected setup, your primary risks are the Local Mirror Registry, Operator Lifecycle Manager (OLM), ImageContentSourcePolicies (ICSP), and the API/Etcd stability under resource pressure.
1. Mirror Registry & Image Resolution Faults
In disconnected environments, all images (including the OpenShift Virtualization operator images) are pulled from a local registry. If this link breaks, the platform cannot scale or recover from node failures.
 * Experiment: Mirror Registry Network Partition
   * Fault: Block traffic from worker nodes to the internal/local mirror registry.
   * Hypothesis: Existing running pods remain stable, but any platform-level reconciliation (like an Operator update or a node restart) will fail. The cluster should report ImagePullBackOff for new infrastructure pods but not crash existing ones.
   * Chaos Toolkit Implementation: Use chaosk8s.actions.network_partition to block the Registry Service IP.
 * Experiment: ICSP/IDMS Corruption
   * Fault: Delete or misconfigure the ImageContentSourcePolicy (ICSP) or ImageDigestMirrorSet (IDMS) that maps registry.redhat.io to your local mirror.
   * Hypothesis: The Cluster Version Operator (CVO) and OLM should immediately alert (Degraded state), as they can no longer resolve platform image manifests.
   * Chaos Toolkit Implementation: Use chaosk8s.actions.delete_custom_resource for the ICSP object.
2. Control Plane & Operator Resilience
These experiments target the "brains" of the platform that manage the Virtualization stack.
 * Experiment: Operator Lifecycle Manager (OLM) Pod Kill
   * Fault: Terminate the catalog-operator or olm-operator pods in the openshift-operator-lifecycle-manager namespace.
   * Hypothesis: OLM is a deployment; the pods should be recreated by the replicaset. The platform's ability to manage the virt-operator should recover within 30 seconds.
   * Chaos Toolkit Implementation: Use chaosk8s.actions.terminate_pods with label app=catalog-operator.
 * Experiment: Etcd Quorum Stress (Disconnected Disk)
   * Fault: Inject high I/O wait or disk pressure on the nodes hosting the Etcd database.
   * Hypothesis: Since disconnected environments often use on-prem storage (ODF/LSO), Etcd must maintain a latency below 10ms. If latency spikes, the API becomes unresponsive.
   * Chaos Toolkit Implementation: Use a stress-ng action on the master nodes to target the /var/lib/etcd mount point.
3. Node-Level Infrastructure Faults
Since OpenShift Virtualization often runs on Bare Metal in disconnected sites, hardware-level failures are critical.
 * Experiment: Kubelet Process Hang
   * Fault: Send a SIGSTOP to the Kubelet process on a worker node.
   * Hypothesis: The OpenShift Machine Config Operator (MCO) should eventually detect the node as NotReady. The platform must trigger a "Node Maintenance" or "Fence" event to protect the workloads.
   * Chaos Toolkit Implementation: Use chaoshsh.actions.run_command to execute kill -STOP <kubelet_pid> via SSH.
 * Experiment: Time Sync (NTP) Drift
   * Fault: Intentionally drift the system clock on a master node in the disconnected network.
   * Hypothesis: OpenShift is extremely sensitive to time sync. If the drift exceeds a threshold, Etcd certificates may become "invalid" (future/past dated), causing the platform to lock up.
   * Chaos Toolkit Implementation: Use chaoshsh.actions.run_command to adjust the chronyd or system time.
